Data Structures That Power Your Database

  • log: an append-only sequence of records. (Log is often used to refer to application logs, where an application outputs text that describes what’s happening)

  • index: keep some additional metadata on the side.

    • speed up read queries
    • slow down writes (append-only storage format best)

Hash Indexes

How it works
  • keep an in-memory hashmap where every key is mapped to a byteoffset in the data file
  • append a new key-value pair to the file : upate the hashmap to reflect the offset of the data you wrote
  • look up a value: use the hashmap to find the offset in the data file, seek to that location, and read the value
  • (segment) each segment has its own hashmap

Requirements
  • all the keys fit in the available RAM (the hash map is kept completely in memory)
  • have a larget number of writes per key, NOT too many distinct keys

Additional problems
  • running out of disk space for data files (here we use append-only files):
    • break the log into segments of a certain size by closing a segment file when it reachs a certain size, and making subsequent writes to a new segment file
    • perform compaction on segments, throw away duplicate keys in the log and keep only the most recent update for each key
    • merge several segments together at the same time as performing the compaction
    • segments are never modified after they have been written, so the merged segment is written into a new file
    • the merging and compaction of frozen segments can be done in background thread, and while it is going on, we can still continue to serve read and write requests as normal, using old segment files
    • after the merging process is complete, we switch read requests to using the new merged segment instead of the old segments – and then the old segment files can simply be deleted

Implementation issues
  • file format: length + raw string
  • deleting records: tombstone
  • crash recovery: a snapshot of each segment’s hash map on disk
  • partially written records: checksums
  • concurrency control: one writer thread, read concurrently

Pros and Cons
  • Pros
  • Cons
  • sequential write operations , faster than random writes
  • concurrency and crash recovery, no overwritten
  • merging old segments avoids the problem of data files getting fragmented over time
  • fit in memory, Not too many keys
  • range queries are not efficient

SSTables and LSM-Trees

SSTable (Sorted String Table}: the sequence of key-value pairs is sorted by key.

LSM-tree:Log-Structured Merge-Tree

Compare log segments with hash indexes
  • Merging segments is simple and efficient, even if the files are bigger than the available memory.
  • Spare index in-memory
  • group several key-value pairs and compress it before writing it to disk to reduce the I/O bandwidth use.

How it works
  • when a write comes in, add it to an in-memory balanced tree data structure (for example, a red-black tree). This in-memory tree is sometimes called a memtable.
  • When the memtable gets bigger than some threshold, write it out to disk as an SSTable file. This can be done efficiently because the tree already maintains the key-value pairs sorted by key. The new SSTable file becomes the most recnet segment of the database. While the SSTable is being written out to disk, writes can continue to a new memtable instance.
  • In order to serve a read request, first try to find the key in the memtable, then in the most recent on-disk segment, then in the next-older segment, etc.
  • from time to time, running merging and compactgion process in the background to combine segment files and to discard overwritten or deleted values

Performance optimizations

Bloom filters: an memory-efficient data structure for approximating the contents of a setl. It can tell you if a key does not appear in the database, and thus saves many unnecessary disk reads for nonexistent keys

size-tiered compaction: newer and smallers SSTables are successively merged into older and larger SSTables

leveled compaction: key range is split up into smaller SSTables and older data is moved into separate levels, which allows the compaction to proceed more incrementally and use less disk space.

B-Trees

Design
  • break the database down into fixed-size blocks or pages, traditionally 4 KB in size
  • read or write one page at a time
  • this design corresponds more closely to the underlying hardware, as disks are also arranged in fixed-size blocks.
  • one page can refer to another page on disk

How it works

  • one page is designated as the root of the B-tree. The page contains several keys and references to child pages.
  • each child is repsonsible for a continuous range of keys, and the keys between the references indicate where the boundaries between those ranges lie.
  • eventually we get down to a page containing individual keys (a leaf page), which either contains the value for each key inline or contains reference to the pages where the values can be found

Reliable
  • overwrite a page on disk with new data, it doesn’t change the location of the page

    include a write-ahead log on disk (an append-only file to which every B-tree modification must be written before it can be applied to the pages of the tree itself)

  • multiple threads are going to access the B-tree at the same time

    protecting the tree’s data structure with latches (lightweight locks)

Optimizations
  • copy-on-write scheme
  • abbreviate the key
  • sequential order of leaf pages on disk
  • additional pointers, leaf page have ref to its sibling pages
  • B-tree variants such as $B^{+}$ trees

Comparing B-Trees and LSM-Trees

  • Advantages of LSM-Trees vs B-Trees
  • Downsides of LSM-Trees vs B-Trees

Higher write throughput
  • LSM-Trees have lower write amplication
    • write amplication: one write to the database results in multiple writes to the disk over the course of the database’s liftime ( SSD concern: can only overwrite blocks a limited number of times before wearing out)
    • LSM-trees rewrite data multiple times due to repeated compaction and merging of SSTables
    • B-trees must write every piece of data at least twice: WAL and tree page itself. Maybe have some other writes.
  • LSM-Trees write compact SSTable files (sequential writes) rather than having to overwrite several pages in the tree (random writes)

Better Compression
  • LSM-Trees compressed better, and produce smaller files on disk
  • B-tree storage engines leave some disk space unused due to fragmentation

Compaction Process
  • LSM-Trees: the compaction process can sometimes interfere with the performance of ongoing reads and writes.
  • B-trees can be more predictable. (Overwrite)

Compaction can’t keep up with the rate of incoming writes
  • LSM-Trees: the disk’s finite write bandwidth needs to be shared between the initial write (logging and flushing a memtable to disk) and the compaction threads running in the background. If write throughput is high and compaction is not configured carefully, it can happen that compaction cannot keep up with the rate of incoming writes.

Transaction Semantics
  • LSM-Trees: have multiple copies of the same key in different segments
  • B-Trees: each key exists in only one place in the index –> transaction isolation is implemented using locks on range of keys

Other Indexing Structures

Storing values within the index

heap file: the place where rows are stored. The value of a key can be a reference to the row stored elsewhere.

clustered index: the extra hop from the index to the heap file is too much of a performance penalty for reads, so it can be desirable to store the indexed row directly within an index.

covering index: a compromise between a clustered index (storing all row data within the index) and a nonclustered index (storing only references to the data within the index), which stores some of a table’s columns within the index

Multi-column indexes

concatenated index: simply combines several fields into one key by appending one column to another

multi-dimensional indexes: for example, R-tree

Full-text search and fuzzy indexes

Transaction Processing or Analytics?

transaction: a group of reads and writes that form a logical unit

Property online transaction processing: OLTP online analytic processing: OLAP
Main read pattern Small number of records per query, fetched by key Aggregate over large number of records
Main write pattern Random-access, low-latency writes from user input Bulk import (ETL) or event stream
Primarily used by End user/customer, via web application Internal analyst, for decision support
What data represents Latest state of data (current point in time) History of events that happened over time
Dataset size Gigabytes to terabytes Terabytes to petabytes

Data Warehousing

Data warehouse: a separate database that analysts can query to their heart’s content, without affecting OLTP operations.

Extract-Transform-Load (ETL): data is extracted from OLTP databases, transformed into an analysis-friendly schema, cleaned up, and then loaded into the data warehouse.

Column-Oriented Storage

row-oriented: all the values from one row of a table are stored next to each other.

column-oriented: don’t store all the values from one row together, but store all the values from each column together instead.

DS-GA 1004 Big Data
  • Column Compression: bitmap encoding, run-length encoding
  • Memory bandwidth and vectorized processing
  • Sort Order in Column Storage
  • Several different sort orders
  • Writing to Column-Oriented Storage
    • all writes first go to an in-memory store, where they are added to a sorted structure and prepared for writing to disk. When enough writes have accumulated, they are merged with the column files on disk and written to new files in bulk.
  • Data Cubes and Materialized Views
    • materialized view: an actual copy of the query results, written to disk
    • virtual view: a shortcut for writing queries
