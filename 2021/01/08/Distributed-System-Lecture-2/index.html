Safety, Liveness, Virtual Clocks

Notes

Safety and liveness

Some reminders from Lecture 1

  • Distributed systems are comprised of many processes (nodes), each of which is isolated from others.

    connected via network, shared-nothing mode

  • processes are connected by an asynchronous network which can arbitrarily delay or drop messages

  • we assume the network is fair: doesn’t look at message content when deciding whether to delay or drop

    arbitrarily

  • we assume processes can fail

Traces/Schedules

trace: a sequence of steps, where at each step exactly one process gets to do something

What can a process do?

  • receives one message
  • does some computation based on its local state and the message
  • sends out one or more messages

Going back to a trace

think of trace as a function:
$$
f: Z^+\rightarrow(P,M)
$$
where $P$ is the set of processes and $M$ is the set of messages.

or think of trace as an infinited array or list:
$$
0 \quad \quad \quad 1 \quad \quad \quad 2,…\
p,m_0 \quad q,m_1 \quad p,m_2, …
$$
where $p,q$ are processes, and $m_1,m_2$ are messages

Prefixes and adding traces

Prefix: a portion of a trace

Protocol correctness as trace properties

  • Safety:

    something never happen in a trace

  • Liveness:

    eventually something good happens

    • Uniform Liveness:

      there is some trace suffix (a part of trace) that can be added to any prefix to achieve the liveness property

    • Absoluate Liveness:

      combining a suffix with any prefix ensures liveness

  • Any program property $P$ can be split into a liveness property $L$ and a safety property $S$ such that $P=L\bigcup S$

Time and its problems

why order events in distributed system?

  • produce a trace, which we can then use to reason about correctness
  • often a requirement for applications

why complicated in ordering events?

  • all information travels at a finite speed, but in networks it is dictated by message delays and latencies

Causality

two events $e_0$ and $e_1$ are causally linked:

  • $e_0$ results in the occurence of $e_1$
  • $e_0$ and $e_1$ are two events in the same process
  • $e_0$ is causally linked to $e_2$, which is causally linked to $e_3$ and so on until $e_n$ which is causally linked to $e_1$

compared the definition with concurrency

Lamport Clocks

Total order

For any two events $e_1,e_2 \in E$, such that $e_1\neq e_2$, we have either $t(e_1)<t(e_2)$ or $t(e_1)>t(e_2)$

On to Lamport clocks

Create a total order among events:

  • all processes send their current clock value in the message
  • when a process receives a message it sets its clock to the maximum of the processes current clock and the clock carried in the message
  • when sending a message the process increments clock value, and then sends a message with update clock value

Cons:

  • total order picks an arbitrary order between events that are causally unrelated
  • the ordering might significantly disagree from what an external observer might have seen

–> solution: build a partial order

Vector clocks

Partial order

For any two events $e_1,e_2 \in E$, such that $e_1\neq e_2$, we have either $t(e_1)<t(e_2)$ or $t(e_1)>t(e_2)$ or $t(e_1)$ can be incomparable to $e_2$.

Vector clocks

  • Clocks are represented as a vector
  • Each component corresponds to a single process
  • Compare by element

Consistent snapshots

Consistent:

  • record state at each process
  • Ensure that the states are recorded in a causally consistent way, i.e. the snapshot should be such that if process $p$ records its state after sending a messgae $m$ to process $q$, then $q$ should only record its state as it exists after it has received message $m$.

“Time, Clocks, and the Ordering of Events in a Distributed System” (Lamport)

Partial Order

  • the events of a process form a sequence, where $a$ occurs before $b$ in this sequence if $a$ happens before $b$.

    that is, a single process is defined to be a set of events with an a priori total ordering

  • (definition) the relation “happen before” $\rightarrow$ :

    • if $a$ and $b$ are events in the same process, and $a$ comes before $b$, then $a \rightarrow b$.
    • if $a$ is the sending message by one process and $b$ is the receipt of the same message by another process, then $a \rightarrow b$.
    • if $a \rightarrow b$ and $b\rightarrow c$, then $a\rightarrow c$.

    or we can say that $a\rightarrow b$ means that it is possible for event $a$ to causally affect event $b$.

  • (current) if $a\nrightarrow b$ and $b\nrightarrow a$, then $a$ and $b$ are concurrent.

    two events are concurrent if neither can causally affect the other.

    For example, $p_3$ and $q_3$ are concurrent in Figure 1.

Logical Clocks

  • clock $C_i$ for each process $P_i$ to be a function which assigns a number $C_i$ to any event $a$ in that process

  • Clock condition: For any events $a, b$: if $a\rightarrow b$ then $C(a)<C(b)$.

    Converse does not hold, since that would imply that any two concurrent events must occur at the same time

  • $\rightarrow$ Holds if two conditions satisfy:

    • if $a$ and $b$ are events in process $P_i$, and $a$ comes before $b$, then $C_i(a)<C_i(b)$
    • if $a$ is the sending of a message by process $P_i$ and $b$ is the receipt of that message by process $P_j$, then $C_j(a)<C_j(b)$

  • To meet 2 conditions above, how do we implement the clock?

    • Condition 1: Each process $P_i$ increments $C_i$ between any two successive events
    • Condition 2:
      • if event $a$ is the sending of a message $m$ by process $P_i$, then the message $m$ contains a timestamp $T_m=C_i(a)$.
      • Upon receiving a message $m$, process $P_j$ sets $C_j$ greater than or equal to its present value and greater than $T_m$

Ordering the Events Totally

  • we simply order the events by the time they occurred. To break ties, we use any arbitrary total ordering $<$ of the processes.

  • $\Rightarrow$ (a way of completing the “happened before” partial ordering to a total ordering): if $a$ is an event in process $P_i$ and $b$ is an event in process $P_j$, then $a \Rightarrow b$ if and only is either:

    • $C_i(a)<C_j(b)$
    • $C_i(a)=C_j(b)$ and $P_i<P_j$

  • if $a\rightarrow b$ then $a\Rightarrow b$.

  • The synchronization for a distributed multiprocess system is specified in term of a State Machine, consisting of a set $C$ of possible commands, a set of $S$ of possible states, and a function $e: C \times S \rightarrow S$. The relation $e(C,S)=S’$ means that executing the command $C$ with the machine in state $S$ causes the machine state to change to $S’$.

  • we will just observe that the entire concept of failure is only meaningful in the context of physical time.

Anomalous Behavior

some external information, how to solve anomalous behavior?

  • explicitly introduce into the system the necessary information about the ordering $\rightarrow$
  • construct a system of clocks which satisfies the “Strong Clock Condition”: For any events $a,b$ in $\Phi$, if $a\longrightarrow b$, then $C(a)<C(b)$. (Here $\longrightarrow$ denote the “happens before” for all system events $\Phi$)

A Total Ordering

introduce Lamport Clock

Problem Statement

  • implement the relation $\rightarrow$ (“happen before”) in which $a\rightarrow b$ iff $a$ must occur before $b$ in all possible event interleavings for this particular distributed computation, i.e. $a$ can causally affect $b$.
  • if $a\nrightarrow b$ and $b\nrightarrow a$ then the two events may have occurred concurrently, i.e. they form part of a consistent slice of the global program state, that is, $a\leftrightarrow b$.

Partial Ordering

Asynchronous Commmunication

timestamps are represented as an array $[c_1,c_2,…c_n]$, with an integer clock value for every process in the network. Let $e_p$ represent an event $e$ executed by process $p$, and $T_{e_p}$ the timestamp array attached to the permanent record of the execution of this event.

For example, the following timestamp $[4,7,6,12]$ attached to an event $x$ in process 2 had local clock value $T_{x_2}[2]=7$ when $x$ was executed, while the last known clock value for process 4 was $T_{x_2}[4]=12$.

Rules about timestamp:

  • Initially all values are zero

  • the local clock vaue is incremented at least once before each atomic event

  • The current value of the entire timestamp array is piggybacked on every outgoing signal

  • upon receiving a signal, a process sets the value of each entry in the timestamp array to be the maximum of the two corresponding values in the local array, and in the piggybacked array received.

    The value corresponding to the sender, however, is a special case and is set to be one greater than the value received (to allow for transit time), but only if the local value is not already greater than that received:

    if local_array[q] <= other_array[q] then

    ​ local_array[q]:=1+other_array[q]

    for i:=1 to n do

    ​ local_array[i]:=max(local_array[i],other_array[i])

  • Values in the timestamp arrays are never decremented.

  • Timestamps attached to the stored records of events as compared as follows:
    $$
    e_p \rightarrow f_q \quad iff \quad T_{e_p}[p]<T_{f_q}[p]
    $$
    event $e_p$ must have occurred before event $f_q$ iff process $q$ has received a signal from $p$ that was sent after, or at the samne time as, the execution of $e_p$.

    so in the figure above we can see that
    $$
    w\rightarrow y \quad \text{since} \quad 2<4\
    l \leftrightarrow v \quad \text{since} \quad (1\nless 0)\and(1\nless 0)\
    d \leftrightarrow z \quad \text{since} \quad (4\nless 2)\and(5\nless 4)\
    b \rightarrow q \quad \text{since} \quad 2<4\
    $$

  • Noticed the “overtaking” situation

    Here we should avoid having $z=[7,4]$. When we set $z$, we should have $[\max(6,5), \max(0,3)+1]$, here we already have $6>5$ so no need to add one to $6$.

Synchronous Communication

In the figure, event $a$ is timestamped earlier than event $b$ although $a$ must occur after $b$.

Here instead of incrementing 1 in the clock, we increment some finite amount of time (for response involved).

how to solve?

  • exchanging timestamps during the communication event, both processes setting their local clock to be equal to the largest value ==>

    receiving process sending back a dummy message with its local clock value to its communication partner, and both processes then adjusting their local clocks accordingly.

  • Rule 3 : during a communication event, the two processes involved exchange timestamp arrays and each element in the local array is set to be the maximum of its old value and the corresponding value in the array received, i.e.

    for i:=1 to n do

    ​ local_array[i]:=max(local_array[i], other_array[i])

Consider a distributed system where the only events are sending and receiving messages. Give pseudocode that each process should run when sending or receiving messages in order to maintain a Vector Clock.

Answer:

Suppose we have $n$ processes $P_1, P_2,…P_i,…P_j,…$ for each process, we have counter $[C_1, C_2, …C_n]$

  1. initially $[C_1=0, C_2=0, …C_n=0]$

  2. SendMessage( $i$, $[C_{1}, C_{2}, …C_{n}]$,​ message)

    $C_{i}=C_{i}+1$

    $[T_{1}, T_{2}, …T_{n}]$=$[C_{1}, C_{2}, …C_{n}]$

    send (message, $[T_{1}, T_{2}, …T_{n}]$)

  3. ReceiveMessage( $j$, $[C_1, C_2, …C_n]$, message, $[T_{1}, T_{2}, …T_{n}]$)

    for $C_k$ in $[C_1, C_2, …C_n]$:

    ​ if $k!=i$:

    ​ $C_k=max(C_k,T_{k})$

    ​ else:

    ​ if $C_{j}<=T_{j}$:

    ​ $C_j=1+T_{j}$
